### protocol-constants  (n_all=8, n_confirmed=8)

Audited C:/Users/stream/HazelRust (the stated source root; note the cwd C:/Users/stream/projects/HazelRust is a DIFFERENT, older checkout and was NOT the audit target). Read hazelcast-core/src/protocol/constants.rs in full and all CP/MAP proxy shadows (proxy/atomic_long.rs, atomic_reference.rs, cp_group.rs, cp_map.rs, fenced_lock.rs, semaphore.rs, countdown_latch.rs, map.rs) plus cluster/cp_management.rs. Cross-checked every suspicious constant against authoritative upstream Hazelcast codecs (CPGroupCreateCPGroupCodec=0x1E0100, CPGroupDestroyCPObjectCodec=0x1E0200, CPSubsystemGetCPGroupIdsCodec=0x220500, MapAddInterceptorCodec=0x011400, MapRemoveInterceptorCodec=0x011500, MapAddPartitionLostListenerCodec=0x011B00, MapRemovePartitionLostListenerCodec=0x011C00, MapPutCodec=0x010100) and the protocol-definitions YAMLs.

PRIOR ADMISSION VERDICT: PARTLY CONFIRMED, PARTLY MITIGATED. The source CP_ATOMIC_LONG_*/CP_ATOMIC_REFERENCE_* constants in constants.rs ARE scrambled exactly as admitted (CP_ATOMIC_LONG_GET=0x090100 is really Apply; CP_ATOMIC_LONG_ADD_AND_GET=0x090500 is really Get; the AtomicRef block is shifted too). BUT in THIS source tree the AtomicLong and AtomicReference proxies define correct LOCAL SHADOW constants and use only those, so the mislabeled source constants are dead for those ops (latent landmine, not a live defect). FencedLock/Semaphore/CountDownLatch/CPMap use the source constants directly, and those particular source values happen to be correct.

REAL, REACHABLE DEFECTS FOUND: (1) The entire CP_SUBSYSTEM_* family is on the WRONG SERVICE (0x1E=CPGroup, only 2 methods) instead of CPSubsystem (0x22). CP_SUBSYSTEM_GET_GROUP_IDS=0x1E0100 is actually CPGroup.createCPGroup; the CP proxies deliberately (mis)use this to create/resolve groups, but cp_management.rs's public get_cp_group_ids/get_cp_group/forceDestroy/getCPMembers/promote/removeCPMember send wrong or nonexistent message types. (2) Four MAP_* constants used by live public map methods carry wrong values that COLLIDE with unrelated operations: MAP_ADD_INTERCEPTOR, MAP_REMOVE_INTERCEPTOR, MAP_ADD_PARTITION_LOST_LISTENER, MAP_REMOVE_PARTITION_LOST_LISTENER. EXCEPTION handling (message type 0) and the spot-checked MAP_GET/PUT/CPMap constants are correct.

### framing-flags  (n_all=4, n_confirmed=4)

Examined hazelcast-core/src/protocol/{constants.rs, frame.rs, client_message.rs, codec.rs} and the CP group-id framing in hazelcast-client/src/cluster/{cp_session.rs, cp_management.rs} and the CP proxy framing (proxy/atomic_long.rs as representative). Cross-checked flag bit values, the request/response header offsets, the message reassembly loop, and the RaftGroupId data-structure framing against the authoritative upstream Hazelcast sources (ClientMessage.java, ClientMessageReader.java, ClientMessageDecoder.java, RaftGroupIdCodec.java) and the published Open Binary Client Protocol spec.

Overall read: The single-frame / unfragmented happy path is wire-correct. Outbound request framing (UNFRAGMENTED=0xC000 on the first frame, IS_FINAL on the last) matches what the server's ClientMessageReader expects (it keys completion solely on IS_FINAL_FLAG). Most flag bit values are correct, and — contrary to the PRIOR ADMISSION — both cp_session.rs::encode_group_id AND the live CP proxy path (atomic_long.rs etc.) DO use proper BEGIN_DATA_STRUCTURE / [seed(8) id(8)] / name / END_DATA_STRUCTURE framing that matches RaftGroupIdCodec exactly (seed@0, id@8). The prior admission does not hold for this revision.

However there are two real defects. (1) CRITICAL: the decoder has NO fragmentation reassembly at all — it conflates the fragment-level BEGIN_FRAGMENT_FLAG (1<<15) with message-begin and terminates on IS_FINAL_FLAG, ignoring fragmentation frames and the 8-byte fragmentation id entirely. The protocol spec mandates that clients MUST support reassembly; fragmentation is at the server's discretion. Any server-fragmented response (large IMap values, entry/key sets, SQL fetch pages) is mis-decoded. (2) MEDIUM (latent): BACKUP_EVENT_FLAG is defined as 1<<11, which is both the wrong value (upstream = 1<<7) and a direct numeric collision with END_DATA_STRUCTURE_FLAG (1<<11); BACKUP_AWARE_FLAG (1<<8) is missing. Currently dormant because no backup-ack/backup-event path is wired up.

### data-serialization  (n_all=7, n_confirmed=7)

Static review of the serialization layer in the audit-target tree C:/Users/stream/HazelRust (hazelcast-core/src/serialization/{traits,data_input,data_output,json,mod}.rs, hazelcast-core/src/protocol/client_message.rs, hazelcast-client/src/proxy/map.rs, hazelcast-client/src/cluster/partition_service.rs), cross-checked against authoritative Hazelcast sources (SerializationConstants, HeapData, HashUtil). NOTE: my cwd was a stale sibling copy (C:/Users/stream/projects/HazelRust/HazelRust); all findings below are quoted from the specified audit root C:/Users/stream/HazelRust.

What is CORRECT: (1) The on-wire Data header layout is right — serialize_value emits [partition_hash i32 BE][type_id i32 BE][payload] with hash placeholder 0; the server recomputes the hash from the payload when the stored hash is 0 (HeapData.hasPartitionHash), so a 0 placeholder is benign. (2) MurmurHash3_x86_32 implementation, seed 0x01000193, mix constants, and the DATA_OFFSET=8 skip used by the normal IMap key path (partition_index_dynamic) all match HeapData.hashCode()/HashUtil. (3) Signed-primitive type-ids i8=-3, bool=-4, i16=-6, i32=-7, i64=-8, f32=-9, f64=-10 and the marker ids PORTABLE=-1, IDENTIFIED=-2, COMPACT=-2 (note: Rust COMPACT_TYPE_ID=-2 conflicts with the real -55, but Compact dispatch is out of this primitive path), JSON const -130 all match upstream. (4) String encoding (i32 UTF-8 byte length + UTF-8 bytes, empty=len 0) matches modern Hazelcast StringSerializer; read_string rejects negative length and non-UTF8. (5) Client UUID encode in auth uses as_u64_pair() (MSB,LSB) written LE — correct ordering. (6) MultiMap routing correctly does compute_partition_hash(skip8(key)).abs()%count.

What is BROKEN (data-corruption / wrong-routing class): a single default type_id() of -11 (STRING) in traits.rs silently mislabels byte arrays, wide unsigned ints, and HazelcastJsonValue; and the four IMap *_with_partition_key affinity methods route using a raw, unbounded, header-included MurmurHash as the partition id (no 8-byte skip, no abs, no % partitionCount).

### swallowed-errors  (n_all=9, n_confirmed=9)

I traced the response path for in-scope ops (IMap get/put/remove/replace/CAS, CP AtomicLong, CPMap, transactions/XA) from the proxy layer through ConnectionManager into the InvocationService, and cross-checked the EXCEPTION-response wire format against upstream Hazelcast (ErrorsCodec EXCEPTION_MESSAGE_TYPE=0, ErrorHolderCodec, ListMultiFrameCodec).

Good news: the re-landed server-exception fix (e6386d2) IS correctly wired for the primary correlation-based path. `InvocationService::invoke` and `invoke_pinned` both call `check_response()` (invocation.rs:270,326), which turns message_type==0 into a typed `HazelcastError::Server`. IMap get/put/remove/replace/CAS (via invoke_on_partition/_mutating -> ConnectionManager::invoke -> InvocationService::invoke), CP AtomicLong and CPMap (via invoke_on_random), and the NON-XA TransactionContext begin/commit/rollback and TxnMap ops (via txn_invoke_at -> invoke_pinned) all route through check_response, so server exceptions surface as Err. The dead helper `txn_invoke` (mod.rs:541, bypasses check_response) is never called.

Bad news: there is a SECOND, parallel invocation path — `ConnectionManager::send_to` + `receive_from` (manager.rs:1610-1634) reading the separate "auth/metadata" connection — that NEVER calls check_response. `Connection::receive()` returns the raw decoded message with no message-type check. Every caller of this path silently turns a server EXCEPTION (type 0) into a "successful" response and then decodes garbage from the error frame. The most severe instance is XA two-phase commit (xa.rs), which is precisely the CBDC-critical path. Several collection/element decoders also swallow per-element deserialize failures, returning silently-short result sets. CPMap's value decoder fails to strip the 8-byte Data header, silently corrupting every CPMap read.

### unsafe-soundness  (n_all=2, n_confirmed=2)

Audited the entire HazelRust workspace (hazelcast-client, hazelcast-core, hazelcast-derive, tests/, benches/) for hand-written unsafe, mem::transmute, from_utf8_unchecked, get_unchecked, MaybeUninit/assume_init, ptr::*, set_len, from_raw_parts, static mut, union, and unsafe impl. Source root: C:/Users/stream/HazelRust (8820-line map.rs matching the line-8668 reference; note the git cwd C:/Users/stream/projects/HazelRust/HazelRust is a stale/different checkout with only 8562 lines). RESULT: there is exactly ONE hand-written unsafe construct in the whole tree — `let key = unsafe { std::mem::zeroed::<K>() };` at hazelcast-client/src/proxy/map.rs:8668. Every other regex hit was a false positive (the word "forget" in XA transaction methods and "fire_and_forget" test names, `std::ptr::eq` which is a safe fn, and a comment in jet/job.rs noting an unsafe impl was removed). The mem::zeroed line is a genuinely bad pattern that produces library-level instant UB for many K, BUT it is gated behind #[cfg(test)] (the `mod tests` at line 5958) AND sits inside a nested helper `fn check_return_type<K,V>` that is never called anywhere in the tree — so the statement is never monomorphized-and-executed for any concrete K. Net runtime CBDC impact: none in production (the code is not even compiled into the library/release artifact), and none even under `cargo test`/MIRI because the line never runs. The real issue is process/claim integrity: the project advertises "zero hand-written unsafe" and there is no #![forbid(unsafe_code)] lint to enforce it, so the convention is only grep-enforced and already has one violation that slipped in. For a CBDC-grade dependency this should be removed and replaced with a compiler-enforced forbid lint.

### panic-safety  (n_all=7, n_confirmed=7)

Examined the network-facing decode path: frame decoder (hazelcast-core/src/protocol/frame.rs), ClientMessage frame assembly and codec (protocol/client_message.rs, protocol/codec.rs), the byte reader (serialization/data_input.rs), and the value deserializers reached from server responses — Compact (serialization/compact/mod.rs, generic_record.rs), Portable (serialization/portable/reader_writer.rs, serializer.rs), serde/json/identified, and the IMap/Cache proxy response decoders (proxy/map.rs, proxy/cache.rs). The frame/codec layer and ObjectDataInput are largely careful: Frame::read_from guards length and never underflows; ObjectDataInput::ensure_remaining gates every primitive read; read_bytes(len) is bounded because ensure_remaining(len) runs before the vec![0u8; len] allocation; read_string rejects negative length. The map proxy fixed-offset readers (decode_int/bool/uuid/entry_view) all length-check before indexing. The systemic defect is attacker-controlled length fields driving Vec::with_capacity / HashMap::with_capacity BEFORE any element is consumed: an i32 length is read from server bytes and cast `as usize`. A negative i32 (e.g. 0xFFFFFFFF) becomes ~usize::MAX → Rust capacity-overflow → process abort; a large positive (e.g. 2e9) → multi-GB allocation → OOM abort. This pattern is pervasive across every Compact and Portable array reader and two IMap/Cache event-journal proxy decoders, and is reachable from a single malformed server/MITM frame. The reader_loop (tokio task) only does panic-safe frame decoding; the dangerous value-deserialization runs inline in the caller's task, so an unwinding panic destroys the in-flight CBDC operation's future (silent loss/hang) and an allocator abort takes down the whole proxy process. The repo's fuzz targets (fuzz_compact_deserialize.rs, fuzz_portable_deserialize.rs) exercise only scalar fields and never an array field, so they do not cover this bug class.

### tls-security  (n_all=8, n_confirmed=7)

Static review of the TLS handshake (connection/connection.rs), its call sites (connection/manager.rs), the WebSocket transport (connection/websocket.rs), and the security module (security/tls_config.rs, authenticator.rs, credentials.rs, kerberos.rs, config.rs SecurityConfig). Source root: C:/Users/stream/HazelRust.

Good news first: the rustls handshake does NOT contain any verification bypass. perform_tls_handshake (connection.rs:308-394) builds ClientConfig::builder().with_root_certificates(root_store).with_no_client_auth()/.with_client_auth_cert(...) and never calls .dangerous(), never installs a custom ServerCertVerifier, and there is no accept-invalid/InsecureSkipVerify-equivalent anywhere in the crate (grep for dangerous/ServerCertVerifier/verify_server_cert = none in code). So the server certificate chain IS validated by default via rustls' default WebPkiServerVerifier (tokio-rustls 0.26 / rustls 0.23.41 per Cargo.lock), and mutual TLS client certs ARE wired (connection.rs:344-374 gated on has_client_auth()). Plaintext is the default (Cargo.toml default=[]; tls is opt-in; NetworkConfig defaults tls.enabled=false), which is the expected posture for an opt-in feature behind a proxy.

The serious problems are in how the security-gate knobs are (not) honored, the identity the cert is verified against, default trust anchors, an alternate transport that ignores TLS config entirely, and cleartext secrets in Debug output. The single most important gate failure: hostname/SAN verification is performed against the member IP literal, not the intended DNS identity, because every caller passes server_name=None and the code defaults the verified name to address.ip(). Combined, these mean the formal "mTLS with full cert + hostname validation" gate is not actually met as configured, and several documented security toggles are silent no-ops.

### retry-idempotency  (n_all=7, n_confirmed=6)

Static analysis of connection/invocation.rs, connection/manager.rs, proxy/map.rs, proxy/atomic_long.rs, proxy/queue.rs, transaction/mod.rs, cluster/cp_session.rs, proxy/fenced_lock.rs and hazelcast-core/src/error.rs. The client has TWO retry layers. (a) Low-level InvocationService::invoke()/invoke_pinned() (invocation.rs:251-341): single attempt, NO retry; on timeout/channel-drop it removes the pending op and returns an error. (b) Manager-level invoke_on_partition_with_retry()/invoke_on_random_with_retry() (manager.rs:1490-1588): a real retry loop gated on `idempotent || redo_operation`, retrying up to invocation_retry_count (default 3) times with a fixed pause, retrying any error where HazelcastError::is_retryable() is true. There is NO operation-level dedup/idempotency token anywhere in the client (confirmed by grep: only Kafka-connector `idempotence` and flake_id local dedup exist). Retries resend the byte-identical message via ClientMessage::clone() — same correlation_id, same embedded payload — and Hazelcast servers do NOT deduplicate non-transactional IMap/Queue/AtomicLong operations by correlation_id. RETRYABILITY (error.rs:295-311) classifies client-side `Timeout(_)` and `Connection(_)` as retryable — these are exactly the AMBIGUOUS failures (request may have been sent and applied; response lost). The retry loop cannot distinguish "send failed before bytes hit the wire" (safe to retry) from "sent, applied server-side, response/connection lost" (unsafe). GOOD NEWS / scoping: with the DEFAULT config (redo_operation=false, config.rs:3099) the catastrophic paths are mostly disarmed: IMap mutations (put/remove/replace/set/putAll) correctly route through invoke_on_partition_mutating (idempotent=false) → max_attempts=0 → single attempt (map.rs:2812-2820); AtomicLong (atomic_long.rs:256-258) and Queue (queue.rs:504-509) route through the base non-retrying manager methods; transactions use invoke_pinned (no retry). The idempotent/mutating split in map.rs is hand-maintained but currently correct (every idempotent=true site is a genuine read: MAP_GET/GET_ALL/CONTAINS_KEY/IS_LOCKED/GET_ENTRY_VIEW). The DANGER is that redo_operation is a documented, user-settable builder flag (config.rs:3007-3014) whose own doc says it will retry put/remove; enabling it turns every ambiguous timeout/disconnect into up to 4 applications of a money mutation with no dedup. Additional hazards: correlation_id reuse across retry attempts can cross-match a late response to the wrong attempt; AtomicLong CP ops carry no invocation UID (unlike FencedLock) so they could never be server-deduped; dead invocation-pool connections are never evicted and reconnect rebuilds only the auth connection.

### test-quality  (n_all=13, n_confirmed=13)

Examined all 36 integration test files under hazelcast-client/tests/ plus the always-running unit tests in hazelcast-client/src (2018 #[test]/#[tokio::test] in src; 165 in proxy/map.rs alone) and hazelcast-core/tests/serialization_proptest.rs. Headline counts: 557 integration test functions, of which 167 are #[ignore]d. SIX entire money-path files run ZERO tests under plain `cargo test`: aggregation_test.rs (7/7 ignored), entry_processor_test.rs (4/4), executor_test.rs (6/6), java_parity_test.rs (38/38), replicated_map_test.rs (11/11), transaction_xa_test.rs (8/8 cluster tests). The other 12 integration files do NOT use #[ignore]; instead every test opens with `if skip_if_no_cluster() { return; }` which makes the test PASS (green), not skip, when no cluster is reachable. The `require_cluster!` macro in common/mod.rs is dead code — used by zero files. Net effect: a developer or CI running `cargo test` with no cluster gets a fully-green suite in which essentially none of the wire-protocol-exercising tests executed. COVERAGE HONESTY: the always-running tests are dominated by local-only checks (config/builder accessors, permission/quorum guards, near-cache LRU bookkeeping, Send/Sync asserts) and SELF-REFERENTIAL serialization (client encodes a frame then the client decodes it — e.g. test_decode_empty_values_response, test_string_frame in map.rs). The Compact proptest proves deserialize(serialize(x))==x against the client's OWN codec, not against real Hazelcast server bytes. There is NO golden-vector test anywhere proving wire compatibility with Hazelcast 5.x, and the partition-routing murmur hash (which decides key->node ownership) has tests that only assert determinism/non-equality/non-zero, never a known-good value, and the routing math diverges from upstream on the i32::MIN edge. SMOKE TESTS on the money path: listener event delivery (map_listener_test, all 6 ignored) increments counters in callbacks but NEVER asserts the counter > 0; topic_stats asserts `received >= 0` (trivially true); entry_processor/aggregation tests put assertions inside the Ok arm and treat Err as a pass (eprintln + return). ISOLATION: integration tests using common::unique_name are process+counter-scoped and reasonably isolated, but java_parity_test.rs (java_parity_test_{name}) and replicated_map_test.rs (e.g. "test-replicated-map-put-get") use FIXED structure names with no PID/UUID, so concurrent runs (or the other process on the shared live cluster) cross-contaminate, and their per-test clear() wipes a peer run's data mid-flight. DETERMINISM: several tests embed real-time sleeps (TTL tests sleep 2-3s; listener tests sleep 500ms+2s) and assume event propagation within that window.
